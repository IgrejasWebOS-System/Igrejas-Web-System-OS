1. Visão Geral do Projeto

O IgrejasWeb System OS é um sistema de gestão eclesiástica multi-tenant de nível Enterprise, projetado para operar múltiplos campos/ministérios de forma isolada e segura. Cada ministério possui dados, usuários, módulos e configurações independentes. O sistema é construído em camadas, da fundação de segurança até os módulos operacionais e, por fim, inteligência analítica.

🏛️
Visão Enterprise

Este roadmap foi ampliado para 17 fases, incorporando todos os módulos de um ChMS Enterprise (Church Management System) completo: multi-campus, financeiro com Fund Accounting, comunicação omnichannel, ministério infantil, governança, BI e API pública. O sistema é arquiteturalmente compatível com white-label para outros segmentos.

Princípios Arquiteturais Fundamentais

  • Multi-tenant por ministry_id: todas as tabelas possuem ministry_id NOT NULL com RLS ativo — dados de um ministério nunca são acessíveis por outro, em nenhuma camada
  • Security by Design / Shift-Left: RLS implementado antes de qualquer módulo operacional. Jamais retrofitado
  • Party Pattern: toda pessoa (membro, aluno, fornecedor, voluntário) é uma Party única com CPF. Papéis são atribuídos via party_roles — sem duplicação de cadastro
  • Feature Flags por Ministério: cada módulo pode ser ativado ou desativado via ministry_modules sem alterar código — comportamento diferente por campo sem deploys
  • RBAC 5 Níveis: N0 Super Master → N1 Admin Campo → N2 Admin Sede → N3 Admin Setor → N4 Usuário Local — com guard em Server Action e em UI
  • Hierarquia de Unidades: CAMPO → SEDE → SETOR → IGREJA → SUB_CONGREGACAO / PONTO_PREGAÇÃO / CÉLULA — SUB_CONGREGACAO é grupo consolidado com liderança definida; PONTO_PREGAÇÃO é estágio inicial informal; evolução: PONTO_PREGAÇÃO → SUB_CONGREGACAO → IGREJA via simples troca de unit_type — com is_headquarters e is_sector_mother para regras de negócio
  • Server Actions como API interna: toda mutação passa por Server Action com assertCanManage(ctx) — nunca acesso direto ao banco pelo client
  • White-label Ready: a arquitetura é agnóstica ao domínio eclesiástico. O mesmo engine de RBAC/multi-tenant serve franquias, clínicas, redes de ensino

Atenção — Party Pattern

Um membro é uma Party com party_role MEMBRO. A mesma pessoa pode ser MEMBRO + VOLUNTÁRIO + ALUNO (Escola Teológica) com um único CPF. Nunca criar tabelas separadas para cada papel. Toda tela que lista pessoas deve usar parties com JOIN em party_roles.

🏛️ Decisões Arquiteturais — Registradas antes da codificação

DEC #1

Modelo de Matrícula — 3 campos distintos

O sistema distingue três tipos de identificação eclesial por motivos pastorais e de migração de dados:

matricula VARCHAR(20) UNIQUE — número definitivo, gerado automaticamente apenas quando o membro passa para ATIVO. Referência principal para todos os módulos.
codigo_provisorio VARCHAR(20) UNIQUE — código temporário para membros ainda não ativados (aguardando batismo, em observação). Substituído pela matrícula ao ativar.
matricula_legado VARCHAR(50) — número do sistema antigo, preservado na importação para garantir continuidade histórica e compatibilidade com registros físicos.
Busca por matrícula, nome ou CPF — com reconhecimento facial planejado como quarto método de busca futuro.
DEC #2

Cargo vs. Função — Dois conceitos distintos e separados

O sistema antigo conflacionava incorretamente estes dois conceitos. Aqui são modelados como entidades separadas:

CARGO — O que o membro É
  • Campo: cargo_id em party_members
  • Único por membro em cada momento
  • Exemplos: MEMBRO, OBREIRO, DIÁCONO, PRESBÍTERO, PASTOR, BISPO
  • Altera-se formalmente via cerimônia eclesiástica
  • Membro sem cargo explícito = "MEMBRO" por padrão
FUNÇÃO — O que o membro FAZ
  • Tabela: party_funcoes
  • Múltiplas simultâneas (n funções)
  • Exemplos: Tesoureira, Líder da Mocidade, Professor EBD, Líder do Coral
  • Vinculada a departamento + escopo geográfico
  • Independente do RBAC do sistema
💡
Exemplo real

Um membro pode ser PRESBÍTERO (cargo) e exercer simultaneamente as funções de Líder da Mocidade no Campo inteiro, Líder do Coral do Setor 01, e Tesoureiro da Igreja Local. Os dois eixos são completamente independentes — ter uma função de liderança departamental NÃO concede acesso N1/N2/N3 ao sistema.

DEC #3

Hierarquia Departamental — Eixo separado do RBAC

A liderança departamental (Mocidade, Coral, Louvor, Dorcas, etc.) opera em uma hierarquia própria que NÃO mapeia diretamente para o RBAC do sistema:

party_funcoes — scope_type
CAMPO — líder regional (atua em todas as igrejas de todo o campo)
SETOR — líder setorial (atua nas igrejas de um setor específico)
IGREJA — líder local (atua em uma única igreja)
SUB_CONGREGACAO — responsável por uma sub-congregação
PONTO_PREGACAO — responsável por um ponto de pregação
CELULA — líder de célula
Um membro pode ter múltiplas entradas em party_funcoes: Líder do Coral no CAMPO + Líder da Mocidade no SETOR 01 + Responsável pela Célula da Igreja 02.
⚠ Restrição de unicidade em party_funcoes UNIQUE constraint em (party_id, department_id, scope_type, unit_id) WHERE is_ativo = TRUE — garante que não haja duas funções idênticas simultâneas ativas para a mesma pessoa no mesmo departamento e escopo.
DEC #4

Tipos de Unidade — SUB_CONGREGACAO e PONTO_PREGAÇÃO como tipos distintos

Dois tipos distintos representam estágios evolutivos de uma congregação em formação. A distinção é pastoral, não só técnica:

PONTO_PREGAÇÃO
  • Estágio inicial, informal
  • Sem local fixo definido
  • Sem liderança formalizada
  • Pai válido: IGREJA ou SUB_CONGREGACAO
SUB_CONGREGACAO
  • Grupo consolidado, estruturado
  • Local físico estabelecido
  • Liderança definida e registrada
  • Pai válido: IGREJA
Evolução registrada no sistema: PONTO_PREGAÇÃO → SUB_CONGREGACAO → IGREJA
Basta alterar o unit_type da unidade existente. Todo histórico de membros, funções e transferências é preservado.

CÉLULA pode ter como pai: IGREJA, SUB_CONGREGACAO ou PONTO_PREGAÇÃO.
Todas as sub-congregações e células são filhas de uma IGREJA — nunca flutuam soltas na hierarquia.
DEC #5

Party Subtypes e Dependentes (filhos)

Todos os tipos de pessoa no sistema são Parties, diferenciados por party_subtype:

MEMBRO_ATIVO — possui matrícula definitiva, situação eclesial ativa
MEMBRO_PROVISÓRIO — aguardando ativação, possui apenas codigo_provisorio
DEPENDENTE — filho menor ou recém-nascido de membro; vinculado via party_dependents
VISITANTE — pessoa não batizada, em acompanhamento pastoral
party_dependents
party_id uuid FK (a criança) · responsible_party_id uuid FK (membro responsável)
relationship ENUM(FILHO, TUTELADO) · data_apresentacao date
data_apresentacao registra o dia em que a criança foi apresentada à Igreja — requisito pastoral para crianças recém-nascidas. Um DEPENDENTE pode ter múltiplos responsáveis (pai e mãe ambos membros).
DEC #6

Tabelas de Lookup — Impede free-text e inconsistências

O sistema antigo permitia texto livre em campos categóricos, gerando "mecânico" vs. "mecanico" vs. "Mecânico" como entradas distintas. Aqui todos os campos categóricos são controlados por lookup tables:

member_cargos — MEMBRO, OBREIRO, DIÁCONO, PRESBÍTERO, PASTOR, BISPO (ecclesial roles)
departments — Mocidade, Coral, Dorcas, Louvor, EBD, Escola Teológica, Missões, etc.
member_funcoes_lookup — Tesoureiro, Secretário, Líder, Sub-líder, Professor, Musico, etc.
member_professions — lista controlada de profissões (sem duplicatas por grafia)
member_schooling — Fundamental, Médio, Superior, Pós-graduação, etc.
member_civil_status — Solteiro, Casado, Divorciado, Viúvo, União Estável
member_genders — Masculino, Feminino
member_financial_status — Ativo (dizimista), Colaborador, Não contribuinte
Todas as lookup tables são por ministério — cada campo pode ter sua lista customizada. Pré-populadas na migration com defaults.

Regras de Negócio — RBAC Financeiro

  • N4 (Usuário Local) só registra receitas da sua unidade — sem acesso a despesas ou aprovações

  • Lançamentos APROVADOS: imutáveis — apenas estorno (novo lançamento) com estorno_de_id

3.5 — Conciliação Bancária via OFX

OFX (Open Financial Exchange) é o formato de extrato bancário exportado pela maioria dos bancos brasileiros. A conciliação compara lançamentos do sistema com o extrato importado e identifica divergências.

6. Regras de Governança do Processo de Desenvolvimento

📜
Governança — Aplica-se a TODOS os projetos desta organização

As regras abaixo não são recomendações. São restrições de processo vinculantes para qualquer sistema desenvolvido por esta equipe, independente de cliente, prazo ou pressão externa.

REGRA #1

Nenhum módulo começa antes do anterior ser totalmente validado

Um módulo está validado quando passou pelo ciclo completo de testes documentado abaixo — não apenas quando a interface funciona visualmente. Avançar antes da validação é proibido mesmo sob pressão de prazo.

O que significa "totalmente validado"
01CRUD completo testado — criar, editar, listar, filtrar, inativar/reativar todos funcionando sem erros
02Isolamento multi-tenant — dado de Campo A não aparece para Campo B em nenhum cenário
03RBAC por nível testado — testado com N0, N1, N3 e N4. Cada nível vê exatamente o que deve ver, nada a mais
04RLS verificado no banco — query direta no Supabase Dashboard com role de N4 retorna zero rows nas tabelas restritas
05Relatórios e PDFs — quando o módulo os gera, verificar dados corretos e layout adequado
06Critério de validação da fase aprovado — o cenário específico descrito neste Roadmap para a fase foi executado com sucesso
Por que esta regra existe Cada módulo neste sistema depende diretamente do anterior. Avançar com fundação inconsistente multiplica o retrabalho em cascata. Uma falha no Party Pattern (Fase 2) descoberta durante a Fase 5 significa reescrever 3 módulos. Uma falha de RLS descoberta em produção é uma violação de dados de todos os ministérios.
REGRA #2

Migrations são imutáveis após execução

Uma migration executada no banco nunca é editada. Toda correção ou ajuste no schema gera uma nova migration com o próximo número sequencial. Isso garante rastreabilidade completa do histórico do banco em qualquer ambiente.

REGRA #3

RLS é implementado antes de qualquer dado operacional

Toda nova tabela tem RLS ativado e políticas definidas na própria migration de criação, nunca retroativamente. Não existe "implementar a segurança depois" — dado sem RLS é dado exposto.

REGRA #4

Este Roadmap é o documento de referência — não memória ou conversa

Toda decisão arquitetural, toda mudança de escopo, todo novo módulo ou regra de negócio deve ser registrada neste Roadmap antes de ser codificada. O documento é atualizado a cada fase concluída e a cada decisão relevante tomada.

🔴 Regra Absoluta — Multi-tenantTodo INSERT deve incluir ministry_id explícito. Todo SELECT deve filtrar por ministry_id. Toda nova tabela deve ter RLS ativo ANTES de qualquer dado ser inserido. Jamais omitir o isolamento multi-tenant em nenhuma camada.
🔴 Regra Absoluta — Ocorrências e Módulos SensíveisRLS por nível (level) para tabelas sensíveis — não apenas por ministry_id. Testar SEMPRE com login de N4 e confirmar que rows retornam vazios via query direta no Supabase Dashboard.
⚠ Nomenclatura de MigrationsSequencial: 005_nome_descritivo.sql. Nunca reeditar migration já executada — criar nova migration de correção.
  • Toda nova tabela deve ter: id uuid PK DEFAULT gen_random_uuid(), ministry_id uuid NOT NULL, created_at timestamptz DEFAULT now(), updated_at timestamptz
  • Trigger automático de updated_at em todas as tabelas que têm este campo
  • Usar .maybeSingle() em vez de .single() para queries que podem retornar null
  • Nunca fazer SELECT * — sempre listar colunas necessárias (performance e segurança)
  • Índices obrigatórios: (ministry_id), (ministry_id, unit_id), (ministry_id, party_id) nas tabelas que referenciam esses campos

Frontend — Next.js 15 App Router

🔴 Regra Absoluta — Server vs. Client ComponentsEvent handlers (onMouseEnter, onClick, onChange, etc.) nunca em Server Components. Extrair para Client Components com "use client" em arquivo separado quando necessário.
⚠ Server ActionsServer Actions são passadas como props bound para Client Components — nunca importadas diretamente dentro de um Client Component (causa erro Turbopack HMR "module factory not available").
  • Server Components para fetch de dados (cookies(), createClient(), queries)
  • Client Components (useState, useTransition, useRef) para toda interatividade
  • revalidatePath() em toda Server Action de mutação — sem stale data
  • Cookies modificados apenas em Server Actions ou Route Handlers — nunca em render de Server Component
  • Usar useTransition + isPending para todos os forms com Server Action — UX de loading sem travar
